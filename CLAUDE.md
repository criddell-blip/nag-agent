## Why this exists

Chris runs a fiber ISP with active buildouts across multiple sites, ongoing federally-funded procurement (Reconnect, BEAD), pole-attachment coordination with utilities (SESD, HL&P), and a field crew that needs daily orchestration. The signal-to-noise across email, ClickUp, and calendar is too high to manage by checking each app — important things slip.

The agent's job is to be **actively annoying in the right places**: nag about overdue tasks, unanswered vendor emails, missed payment reviews, and meeting prep — and shut up about things that don't matter. Companion dashboard gives the visual overview when needed.

**Explicit design principles:**

- Active nag > passive dashboard. The dashboard supports the nag, not the other way around.
- Learning loop with explicit override — never silently change behavior, always surface "want me to mute X?" for approval.
- **Keyword/regex classification only for v1** — every starter rule is solvable with sender allowlists, subject regex, status checks, and date math. No LLM dependency. May add Claude API later if specific rules prove too brittle to express as regex.
- Start narrow (SESD + ClickUp), expand once patterns are proven.
- **Loud by default** — critical alerts re-nag hourly (gated by 22:00–06:00 MT quiet hours). Tune down if it's too much; opposite is the harder direction.
- **People-table-driven matching** — rules reference `people.priority_tier` / `role_category` / `org` instead of hardcoded names, so updating who matters is a row edit, not a rule rewrite.

## Architecture

Three layers, each with a clear responsibility:

```
┌─────────────────────────────────────────────────────────┐
│  SOURCES (read-only)                                    │
│  Gmail · ClickUp · Google Calendar                      │
└─────────────────────────────┬───────────────────────────┘
                              │ ingest (15 min cron)
                              ▼
┌─────────────────────────────────────────────────────────┐
│  SUPABASE                                               │
│  ─ events (normalized from all sources)                 │
│  ─ rules (JSON conditions + actions)                    │
│  ─ alerts (generated by rule engine)                    │
│  ─ alert_dispositions (user feedback for learning)      │
│  ─ learned_patterns (aggregated for rule suggestions)   │
└───────────────┬─────────────────────────────┬───────────┘
                │                             │
                ▼                             ▼
┌────────────────────────────┐  ┌────────────────────────┐
│  SLACK BOT                 │  │  DASHBOARD (React)     │
│  (nag layer)               │  │  (visual layer)        │
│  ─ DM critical alerts      │  │  ─ Calendar pane       │
│  ─ Interactive buttons     │  │  ─ Phase tracker pane  │
│  ─ Snooze/ack/dismiss      │  │  ─ Alert queue pane    │
│  ─ Daily morning digest    │  │  ─ Realtime sync       │
└────────────────────────────┘  └────────────────────────┘
```

**Data flow:**

1. Cron triggers ingest function every 15 min.
2. Ingest pulls deltas from Gmail/ClickUp/Calendar, normalizes into `events`.
3. Rule engine evaluates open events against active `rules`, generates `alerts` with severity.
4. Slack bot picks up new alerts, sends DMs with interactive buttons.
5. User responds in Slack → disposition logged → eventually surfaces as a "want me to mute this?" suggestion.
6. Dashboard subscribes to Supabase Realtime, reflects current state.

## Database schema (draft)

> First migration should establish these tables. Names and columns are starting points — adjust as the design matures.

```sql
-- Unified event stream from all sources
create table events (
  id uuid primary key default gen_random_uuid(),
  source text not null check (source in ('gmail', 'clickup', 'gcal')),
  external_id text not null,        -- Gmail thread ID, ClickUp task ID, Calendar event ID
  external_url text,
  event_type text not null,         -- 'email_received', 'task_due', 'meeting_starting', etc.
  subject text,
  body_excerpt text,
  sender text,                      -- email address or user ID
  participants text[],
  occurred_at timestamptz not null,
  due_at timestamptz,
  status text,                      -- task status, email read/unread, etc.
  raw_metadata jsonb,               -- whatever the API returned
  ingested_at timestamptz default now(),
  unique (source, external_id)
);

create index events_occurred_at_idx on events (occurred_at desc);
create index events_due_at_idx on events (due_at) where due_at is not null;
create index events_source_idx on events (source);

-- Rules: declarative conditions + actions
create table rules (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  description text,
  enabled boolean not null default true,
  -- JSON shape: { source, sender_match, subject_match, body_match, age_threshold, status_match, ... }
  conditions jsonb not null,
  -- JSON shape: { severity, channel, snooze_default, escalation, ... }
  actions jsonb not null,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- Alerts: generated by rule engine, consumed by Slack bot + dashboard
create table alerts (
  id uuid primary key default gen_random_uuid(),
  rule_id uuid references rules(id),
  event_id uuid references events(id),
  severity text not null check (severity in ('info', 'warn', 'critical')),
  title text not null,
  body text,
  context_links jsonb,              -- [{label, url}, ...] to email, task, etc.
  status text not null default 'open' check (status in ('open', 'acknowledged', 'snoozed', 'dismissed', 'done')),
  snooze_until timestamptz,
  created_at timestamptz default now(),
  last_notified_at timestamptz,     -- for daily re-nag throttling
  closed_at timestamptz
);

create index alerts_status_idx on alerts (status) where status = 'open';
create index alerts_snooze_idx on alerts (snooze_until) where status = 'snoozed';

-- Dispositions: user feedback, drives the learning loop
create table alert_dispositions (
  id uuid primary key default gen_random_uuid(),
  alert_id uuid not null references alerts(id),
  action text not null check (action in ('acknowledged', 'snoozed', 'dismissed', 'marked_done', 'false_positive')),
  snooze_duration_hours int,
  user_note text,
  created_at timestamptz default now()
);

-- Integration credentials (refresh tokens, etc.)
create table integrations (
  id uuid primary key default gen_random_uuid(),
  service text not null unique,     -- 'gmail', 'clickup', 'gcal', 'slack'
  encrypted_credentials text,       -- via supabase_vault
  last_sync_at timestamptz,
  sync_cursor text,                 -- e.g. Gmail history ID, ClickUp page token
  active boolean default true
);

-- Aggregated learning signals
create table learned_patterns (
  id uuid primary key default gen_random_uuid(),
  pattern_type text not null,       -- 'sender_dismissed_often', 'rule_too_noisy', etc.
  details jsonb,
  suggested_action text,            -- e.g. "mute sender X" or "tighten rule Y"
  status text default 'pending' check (status in ('pending', 'accepted', 'rejected')),
  surfaced_at timestamptz default now()
);

-- People the rule engine cares about (added in migration 0003)
-- Each row = one email address. Same human at multiple addresses = multiple rows.
create table people (
  id uuid primary key default gen_random_uuid(),
  email text not null unique,
  name text,
  org text,
  role text,
  role_category text not null check (role_category in (
    'internal','vendor','utility','partner_isp','government','contractor',
    'engineering','legal','automated','sales','family','applicant','customer','other'
  )),
  priority_tier text not null default 'normal' check (priority_tier in (
    'critical','high','normal','low','noise'
  )),
  notes text,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

**Rule condition shape (the JSON the matcher will interpret):**

| Field | Meaning |
|---|---|
| `source` | `'gmail'` / `'clickup'` / `'gcal'` |
| `sender_priority_in` | array of `people.priority_tier` values (e.g. `["critical","high"]`) |
| `sender_role_in` | array of `people.role_category` values |
| `sender_role_not_in` | exclude these role categories (e.g. exclude `internal` from external escalation rules) |
| `sender_org_in` | array of `people.org` names |
| `sender_in` | array of specific emails (overrides people-table lookup) |
| `sender_regex` | POSIX regex against `events.sender` (for ad-hoc patterns) |
| `subject_regex` | POSIX regex against `events.subject` |
| `body_regex` | POSIX regex against `events.body_excerpt` |
| `has_attachments` | bool (Gmail only) |
| `age_min_hours` / `age_max_hours` | event age window |
| `due_within_hours` / `past_due` | task / calendar event due-date checks |
| `status_in` | array of event statuses |
| `status_change_to` | event status flipped to this value |
| `exclude_noise_senders` | bool — auto-skip `people.priority_tier='noise'` |

**Rule action shape:**

| Field | Meaning |
|---|---|
| `severity` | `'info'` / `'warn'` / `'critical'` |
| `title_template` | human-readable title; `{subject}` / `{sender}` substituted |
| `re_nag` | `'hourly'` (critical only, during business hours) / `'daily'` (morning) / `'none'` |
| `escalate_after` | bump severity to critical after N days unhandled |
| `auto_close_event` | bool (e.g. payment confirmation auto-marks related task done) |

**Quiet hours:** globally 22:00–06:00 MT, enforced by the matcher regardless of `re_nag` setting. Critical alerts that fire overnight batch into a 06:00 wake-up DM.

## Conventions & gotchas

> Pulled from hard-won lessons on FiberLog. Apply these from the start.

**Supabase joins always return related tables as arrays**, even for single objects. Always normalize:

```js
const sender = Array.isArray(event.sender) ? event.sender[0] : event.sender;
```

**Edge Functions for everything cron + integration.** Don't put long-running ingest in the React app. Pattern:

- `ingest-gmail` — Scheduled Function, every 15 min, fetches deltas via Gmail history API
- `ingest-clickup` — Scheduled Function, every 15 min, fetches updated tasks since last sync
- `ingest-gcal` — Scheduled Function, every 15 min, fetches events in next 14 days
- `evaluate-rules` — runs after each ingest, fires alerts
- `slack-notify` — triggered by inserts to `alerts`, formats and sends Slack DMs
- `slack-interactions` — handles button clicks from Slack, updates alert status

**OAuth refresh tokens go in `supabase_vault`**, not in plain columns. Use `vault.secrets` and reference by name.

**Idempotent ingest.** Use the `unique (source, external_id)` constraint and `upsert` so re-runs don't double-insert events.

**Throttle re-nags.** A critical alert that's still open should re-ping once per morning, not every 15 min. Use `last_notified_at` to gate.

**Phase number extraction.** SESD emails reference phases like "PH 4.2" or "Phase 3.1" — write a regex helper early and reuse it everywhere. This is how email → ClickUp task linkage happens.

**Sender allowlist for critical.** Hardcode initial trust list (Ryan/Brook/Heather/Mike/Melinda) in seed data. Don't try to learn it from scratch.

## Locked decisions (May 2026)

These were the §4 open decisions from the build plan; all settled before scaffolding:

- **Primary nag channel:** Slack only. UBB workspace, Chris is admin (no install blockers).
- **Dashboard location:** Standalone app. Separate React+Vite project, own GitHub Pages deploy. FiberLog stays single-purpose.
- **Classification approach:** Keyword/regex only for v1. Every starter rule is rule-expressible. No Claude API account, no LLM costs. Revisit only if a specific rule proves too brittle.
- **Learning aggressiveness:** Suggest only — never silently mute or change a rule without Chris's approval. (Default; revisit at Phase 5.)
- **v1 access:** Solo. Multi-user comes after the patterns are proven. (Default; revisit when team interest emerges.)
